External APIs are messy: spotty webhooks, rate limits, undetectable deletes, cursor management, memory walls on large datasets. Teams spend weeks on plumbing and still miss records.

Nango handles all of it. You define a function that fetches what you need; Nango runs it on a schedule (or on webhook), checkpoints progress, detects adds/updates/deletes, and pushes a webhook to your app when new data lands. Pick from **thousands of pre-built sync functions** or write your own.

**Common patterns:**

- Sync CRM records (HubSpot, Salesforce, Attio…) — contacts, companies, deals
- Sync files from drives (Google Drive, SharePoint, Box…)
- Sync call transcripts (Gong, Fathom, Zoom…)
- Feed RAG pipelines with fresh customer data
- Fire per-record triggers when something changes upstream

<Tip>
For agentic / on-demand tool calling against external APIs (no local replication needed), use [Tool calling for AI agents](/implementation-guides/use-cases/tool-calling) instead.
</Tip>

## How Nango fits in

<Frame caption="Data flows from the external API through the records cache to your application.">
  ![](/images/diagrams/nango-syncs-diagram.png)
</Frame>

1. **You configure** an integration in Nango (e.g. Salesforce) and pick which sync functions to enable — from thousands of [pre-built templates](https://www.nango.dev/templates) or your own custom code.
2. **Your user authorizes** via a Connect link. Nango handles OAuth, stores tokens, and refreshes them as they expire.
3. **Nango runs your function** on a schedule (or on webhook), with retries, rate-limit handling, and pagination. It writes records to Nango's records cache.
4. **Nango detects changes** (adds/updates/deletes) by comparing payload hashes, and sends a webhook to your app whenever fresh data is available.
5. **Your app reads the delta** from the records cache via cursor — never re-processing what you've already seen.

## Why Nango

- **Built-in change detection.** Nango hashes every record and surfaces only what changed since your last cursor. No diffing on your side.
- **Resilient incremental syncs via [checkpoints](/implementation-guides/use-cases/syncs/checkpoints).** Save progress mid-run; resume from where the last execution stopped on failure or interruption — even on million-record datasets.
- **Real-time when you need it.** Combine [webhooks from external APIs](/implementation-guides/use-cases/syncs/realtime-syncs) with periodic polling for a reliable stream of changes that's resilient to provider outages.
- **Detect deletions** even on APIs that don't expose them, via [trackDeletes](/implementation-guides/use-cases/syncs/deletion-detection) or a recently-deleted endpoint.
- **Multi-tenant via [variants](/implementation-guides/use-cases/syncs/sync-partitioning).** Run multiple independent instances of the same sync per connection, each with its own filter or schedule.
- **Two-way sync.** Combine sync functions with [action functions](/implementation-guides/use-cases/actions) to write changes back to the external API.
- **Encrypted, observable, scoped per user.** Records are encrypted at rest and in transit; every run is logged; each function executes in the context of an end-user [Connection](/guides/primitives/auth#overview).

<Step title="Create the sync file">

<Tip>
    If you prefer to automate this, follow the [Build with AI guide](/guides/primitives/functions#guide).

In your `nango-integrations` folder, create the file for your new sync function.

Sync files should be within a `syncs` folder, which is nested under the integration's folder.

For example, if you want to create a new sync to sync `contacts` from `salesforce`, your structure should look like this:

```
nango-integrations/
├── .nango
├── .env
├── index.ts
├── package.json
└── salesforce # this is the integration id and must match an integration id in your Nango dashboard
    └── syncs/
        └── salesforce-contacts.ts # this is the name of your sync
```

In your sync file, paste the following scaffold:

```ts salesforce-contacts.ts lines
import { createSync } from 'nango';
import * as z from 'zod';

const MyObject = z.object({
  id: z.string(),
  first_name: z.string(),
  last_name: z.string(),
  email: z.string(),
});

export default createSync({
    description: `<Description of your sync>`,
    version: '1.0.0', // Version, increment it when you release a new version
    frequency: 'every hour', // Default sync interval
    autoStart: true, // Should the sync start immediately when a new connection is created?
    trackDeletes: true, // detect deletes? See separate implementation guide
    // Optional: define a checkpoint schema to save progress and avoid re-fetching all data
    checkpoint: z.object({
        lastModifiedISO: z.string(),
    }),
    models: {
        MyObject: MyObject,
    },
    exec: async (nango) => {
        // Integration code goes here.
    },
});
```

Also import your new sync file in your `index.ts` file:

```typescript index.ts
import './salesforce/syncs/salesforce-contacts';
```

</Step>

<Step title="Implement your sync">

In the `exec` method, implement the logic of your sync. Edit `MyObject` to contain the properties you need.

The following can help you with your implementation:
- [`nango` object reference](/reference/functions) to understand the SDK methods available in functions
- [Our integration templates repo](https://www.nango.dev/templates) has thousands of examples of sync and action functions implemented by Nango
- [Leveraging AI agents guide](/guides/primitives/functions#guide) to build sync functions with Claude Code, Cursor & other AI agents

Example implementation of the Salesforce contacts sync:

```ts salesforce-contacts.ts lines
import { createSync } from 'nango';
import * as z from 'zod';

const SalesforceContact = z.object({
  id: z.string(),
  first_name: z.string(),
  last_name: z.string(),
  email: z.string(),
  account_id: z.string().nullable(),
  last_modified_date: z.string(),
});

const sync = createSync({
  description: `Fetches contacts from Salesforce`,
  version: '1.0.0',
  frequency: 'every hour',
  autoStart: true,
  // Define checkpoint schema to track sync progress
  checkpoint: z.object({
    lastModifiedISO: z.string(),
  }),
  models: {
    SalesforceContact: SalesforceContact,
  },
  exec: async (nango) => {
    // Get checkpoint to resume from where we left off
    const checkpoint = await nango.getCheckpoint();
    const query = buildQuery(checkpoint?.lastModifiedISO);

    await fetchAndSaveRecords(nango, query);

    await nango.log('Function run completed!');
  },
});

export default sync;
type NangoSyncLocal = Parameters<(typeof sync)['exec']>[0];

function buildQuery(lastModifiedISO?: string): string {
    let baseQuery = `SELECT Id, FirstName, LastName, Email, AccountId, LastModifiedDate FROM Contact`;

    if (lastModifiedISO) { // Only fetch data modified since last sync
        baseQuery += ` WHERE LastModifiedDate > ${lastModifiedISO}`;
    }
    baseQuery += ` ORDER BY LastModifiedDate ASC`;

    return baseQuery;
}

async function fetchAndSaveRecords(nango: NangoSyncLocal, query: string) {
    let endpoint = '/services/data/v53.0/query';

    while (true) {
        const response = await nango.get({
            endpoint: endpoint,
            params: endpoint === '/services/data/v53.0/query' ? { q: query } : {}
        });

        const mappedRecords = mapContacts(response.data.records);

        await nango.batchSave(mappedRecords, 'SalesforceContact'); // Saves records to Nango cache.

        // Save checkpoint for next run
        await nango.saveCheckpoint({ lastModifiedISO: mappedRecords[mappedRecords.length - 1].last_modified_date });

        if (response.data.done) {
            break;
        }

        endpoint = response.data.nextRecordsUrl;
    }
}

function mapContacts(records: any[]): SalesforceContact[] {
    return records.map((record: any) => {
        return {
            id: record.Id as string,
            first_name: record.FirstName,
            last_name: record.LastName,
            email: record.Email,
            account_id: record.AccountId,
            last_modified_date: record.LastModifiedDate
        };
    });
}
```

In this integration function, the following Nango utilities are used:
- `await nango.getCheckpoint()` retrieves the saved checkpoint (returns `null` on first run)
- `await nango.saveCheckpoint()` saves progress to resume from on next run
- `await nango.batchSave()` to persist external data in Nango's cache
- `await nango.get()` to perform an API request (automatically authenticated by Nango)
- `await nango.log()` to write custom log messages

Depending on your needs, you may want to dive deeper into:

- [Checkpoints](/implementation-guides/use-cases/syncs/checkpoints) — make your sync incremental and resilient to failures.
- [Deletion detection](/implementation-guides/use-cases/syncs/deletion-detection) — surface deletes from the external API.
- [Real-time syncs](/implementation-guides/use-cases/syncs/realtime-syncs) — combine webhooks with polling for sub-second freshness.
- [Records cache](/implementation-guides/use-cases/syncs/records-cache) — how Nango stores records and exposes deltas via cursors.
- [Sync partitioning](/implementation-guides/use-cases/syncs/sync-partitioning) — run multiple variants of the same sync per connection.

**For higher reliability, use cursors.** Webhook notifications can be missed, and relying solely on the webhook payload to fetch modified records can cause you to miss some updates.

A more reliable way of keeping track of how far you've synced records (for each connection & function combination) is to rely on record **cursors**.

Each record comes with a synchronization cursor in `_nango_metadata.cursor`. Nango uses cursors internally to keep a chronological list of record modifications.

Each time you fetch records, you should store the `cursor` of the last record you fetched to remember how far you've synced (for each connection & sync combination).

The next time you fetch records, pass in the cursor of the last-fetched record to only receive records modified after that record:

        ```bash
        --header 'Authorization: Bearer <ENVIRONMENT-SECRET-KEY>' \
        --header 'Provider-Config-Key: <providerConfigKey-in-webhook-payload>' \
        --header 'Connection-Id: <connectionId-in-webhook-payload>' \
        --data-urlencode 'model=<model-in-webhook-payload>' \
        --data-urlencode 'cursor=<cursor-of-last-fetched-record>' \
        ```
    </Tab>
    <Tab title="Node SDK">
        ```ts
        import { Nango }  from '@nangohq/node';

        const nango = new Nango({ secretKey: '<ENVIRONMENT-SECRET-KEY>' });

        const result = await nango.listRecords({
            providerConfigKey: '<providerConfigKey-in-webhook-payload>',
            connectionId: '<connectionId-in-webhook-payload>',
            model: '<model-in-webhook-payload>',
            cursor: '<cursor-of-last-fetched-record>'
        });
        ```
    </Tab>
</Tabs>

So, the overall logic for cursor-based synchronization should be:
1. Receive a webhook notification from Nango
2. Query your database for the cursor of the last-fetched record
3. Fetch the modified records (passing the cursor)
4. Store the modified records
5. Store the last-fetched record cursor

</Step>

## Two-way syncs

The hard part of staying in sync with an external API is the read side — pulling fresh data, deduping, detecting changes, handling deletes. That's what sync functions solve.

Writing data **back** to the external API is comparatively easy and doesn't need polling: you do it on demand from your app via [action functions](/implementation-guides/use-cases/actions), which run in real time and return the result synchronously. Pair the two for true two-way sync — read with sync functions, write with action functions.

## Sync efficiency

Two factors dominate sync performance and cost: how much data each run transfers, and how often runs happen.

### Full vs. incremental

A **full sync** re-fetches every record from the external API on every run. Simple and works for any API, but cost scales with total dataset size — you re-pay for the same data forever.

An **incremental sync** only fetches what changed since the last run, by passing a checkpoint (a timestamp, cursor, or page token) back to the API. The first run is still expensive, but every run after only processes the delta.

Whether you can run incrementally depends on the API. Incremental works when the API supports filtering by modification timestamp, sorting by `updated_at`, or exposing a "since cursor". If the API only lets you list everything, you're stuck with full syncs.

**Incremental is dramatically more efficient than full** — fewer API calls, lower bandwidth, less compute, faster runs, lower cost — and it's safer because checkpoints make runs resumable on failure. Use incremental whenever the API supports it.

### Practices for efficient syncs

- **Run only as often as the business needs.** A 30-minute sync is six times cheaper than a 5-minute sync. Pick the lowest cadence that meets your freshness requirements — especially for full syncs.
- **Make full syncs incremental wherever possible.** Even a coarse `since` filter (last 7 days) cuts cost dramatically vs. fetching everything.
- **For full syncs that genuinely can't go incremental, evaluate webhooks instead.** Many APIs that don't support `since` filtering do support webhooks — Nango can process them directly inside a sync function. [Reach out](https://nango.dev/slack) if you'd like help wiring this up.
- **Keep output models small.** Drop fields you don't need before calling `batchSave()` — the records cache stores everything you save, and smaller records mean less compute, bandwidth, and storage downstream.
- **Review API interactions and function logic for inefficiencies.** A duplicated request, an unnecessary join, or a `for` loop that could be a single bulk call adds up fast at scale.

### Avoiding memory overuse

Sync functions run on VMs with fixed resources. The most common cause of memory-related failures is accumulating records in memory instead of saving them as you go.

Don't do this:

```ts
exec: async (nango) => {
    const checkpoint = await nango.getCheckpoint();
    const allRecords: any[] = [];

    while (nextPage) {
        const res = await nango.get({ endpoint: '/contacts', params: { since: checkpoint?.lastModifiedISO, cursor: nextPage } });
        allRecords.push(...res.data.records); // Accumulates in memory
    }

    await nango.batchSave(mapContacts(allRecords), 'Contact'); // Saves everything at once
};
```

Instead, save records and checkpoint after each page:

```ts
exec: async (nango) => {
    const checkpoint = await nango.getCheckpoint();

    while (nextPage) {
        const res = await nango.get({ endpoint: '/contacts', params: { since: checkpoint?.lastModifiedISO, cursor: nextPage } });

        const contacts = mapContacts(res.data.records);
        await nango.batchSave(contacts, 'Contact');
        await nango.saveCheckpoint({ lastModifiedISO: contacts[contacts.length - 1].lastModified });

        nextPage = res.data.nextCursor;
    }
};
```

This releases each page from memory after saving, keeping usage constant regardless of dataset size. Monitor memory consumption in the **Logs** tab of the Nango dashboard.

### Filtering unnecessary data

Filter as early as possible — using filters provided by the external API or by discarding records before calling `batchSave()`. This keeps the external system as the source of truth: if you need additional fields later, edit your function and trigger a sync with `reset: true` to backfill.

You can also use [connection metadata](/reference/sdks/node#get-connection-metadata) to implement per-customer filters in your sync functions.

## Data retention

<Warning>
**Data retention policies**

Nango automatically manages data retention for synced records:
- **Records not updated for 30 days**: Payload is pruned (metadata like record ID and sync state remain for change detection)
- **Syncs not executed for 60 days**: All records from that sync are permanently deleted

**Best practice**: Fetch records from Nango promptly after receiving webhook notifications and store them in your own system. Don't use Nango's cache as your primary long-term data store.

[Learn more about data retention policies](/guides/platform/security#synced-records-retention)
</Warning>
