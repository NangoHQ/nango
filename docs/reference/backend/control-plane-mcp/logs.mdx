The Control Plane MCP server provides two tools for investigating activity in the authenticated Nango environment. Use them to find relevant operations, then inspect the messages for a specific operation.

## Authentication

Both tools require a Nango API key with the `environment:logs:read` scope. See the [Control Plane MCP overview](/reference/backend/control-plane-mcp/overview) for the server URL and authentication setup.

## `logs_list_operations`

Lists Nango log operations newest first. An operation is a top-level execution record for activity such as a sync, action, authentication flow, webhook, or proxy call.

By default, the tool returns up to 25 operations created in the last 24 hours.

### Arguments

| Argument       | Type             | Description                                                                                    |
| -------------- | ---------------- | ---------------------------------------------------------------------------------------------- |
| `search`       | string           | Search operation messages. Maximum 256 characters.                                             |
| `limit`        | integer          | Maximum operations to inspect. From 1 to 500; defaults to 25.                                  |
| `cursor`       | string or `null` | Cursor returned by a previous call.                                                            |
| `states`       | string[]         | Filter by `waiting`, `running`, `success`, `failed`, `timeout`, or `cancelled`.                |
| `operations`   | object[]         | Filter by operation type and, optionally, action. Maximum 20 filters.                          |
| `integrations` | string[]         | Filter by integration name. Maximum 20 values.                                                 |
| `connections`  | string[]         | Filter by connection name. Maximum 20 values.                                                  |
| `syncs`        | string[]         | Filter by sync name. Maximum 20 values.                                                        |
| `period`       | object           | ISO 8601 date range with required `from` and optional `to`. `to` defaults to the current time. |

The `operations` argument accepts these type and action combinations:

| Type      | Actions                                                                                                            |
| --------- | ------------------------------------------------------------------------------------------------------------------ |
| `action`  | `run`                                                                                                              |
| `sync`    | `pause`, `unpause`, `run`, `request_run`, `request_run_full`, `cancel`, `init`, `create_variant`, `delete_variant` |
| `proxy`   | `call`                                                                                                             |
| `events`  | `post_connection_creation`, `pre_connection_deletion`, `validate_connection`                                       |
| `auth`    | `create_connection`, `refresh_token`, `post_connection`, `connection_test`                                         |
| `admin`   | `impersonation`                                                                                                    |
| `webhook` | `incoming`, `forward`, `sync`, `connection_create`, `connection_refresh`                                           |
| `deploy`  | `prebuilt`, `custom`                                                                                               |

Omit `actions` to include every action for that type.

Example arguments for finding failed sync executions:

```json
{
    "states": ["failed"],
    "operations": [{ "type": "sync", "actions": ["run"] }],
    "period": { "from": "2026-07-12T00:00:00.000Z" },
    "limit": 50
}
```

### Result

The tool returns:

- `operations`: matching operation records. Each record includes its ID, state, operation type and action, timestamps, and related integration, connection, or function identifiers when available.
- `pagination.total`: number of matching operations. When `search` is set, this is the number of matches in the inspected page.
- `pagination.cursor`: cursor for the next page, or `null` when there are no more pages.

<Note>
    When `search` is set, `limit` is the maximum number of operations whose messages are inspected, not a guaranteed number of matches. A page can contain fewer
    or no matching operations and still return a cursor.
</Note>

## `logs_get_operation`

Gets one operation and a page of its log messages. Messages are returned newest first.

### Arguments

| Argument          | Type             | Description                                                                                    |
| ----------------- | ---------------- | ---------------------------------------------------------------------------------------------- |
| `operationId`     | string           | ID returned by `logs_list_operations`. Required.                                               |
| `messages`        | object           | Optional message pagination and filtering settings.                                            |
| `messages.limit`  | integer          | Maximum messages to return. From 1 to 500; defaults to 25.                                     |
| `messages.cursor` | string or `null` | Cursor returned by a previous call.                                                            |
| `messages.search` | string           | Search operation messages. Maximum 100 characters.                                             |
| `messages.period` | object           | ISO 8601 date range with required `from` and optional `to`. `to` defaults to the current time. |

Example arguments:

```json
{
    "operationId": "<OPERATION-ID>",
    "messages": {
        "search": "rate limit",
        "limit": 100
    }
}
```

### Result

The tool returns:

- `operation`: the requested operation record.
- `messages`: log messages for the operation. Each message includes its ID, level, type, message, timestamp, and related error, HTTP request, HTTP response, retry, or metadata details when available.
- `pagination.total`: total messages matching the filters.
- `pagination.cursor`: cursor for the next message page, or `null` when there are no more pages.

Pass each returned cursor back to the same tool with the same filters to retrieve the next page.
